## Commands

```bash
npm run dev       # Start dev server with tsx watch (hot reload)
npm run build     # Compile TypeScript → dist/
npm start         # Run compiled server from dist/
npm run prisma    # Regenerate Prisma client after schema changes
npm run bs        # Build then start (production-like)
```

No linting or test runner is configured.

After any `prisma/schema.prisma` change, run `npm run prisma` to regenerate the client. If adding typed SQL queries, place `.sql` files in `prisma/sql/` and regenerate.

## Architecture

Express.js REST API (TypeScript, ES modules) with a PostgreSQL database via Prisma ORM.

**Request flow:** `routes/` → `controllers/` → `services/` (for complex ops) → Prisma

- **Routes** (`src/routes/`): Thin Express routers. One file per domain (assets, arrivals, departures, transfers, holds, invoices, models, organizations, constants).
- **Controllers** (`src/controllers/`): Parse requests, validate with Zod schemas, call services or Prisma directly, return responses.
- **Services** (`src/services/`): Business logic for operations that require multiple steps or DB calls. Currently only `arrivalService.ts` exists — it generates sequential barcodes and arrival numbers using a PostgreSQL `get_next_sequence()` function, then creates the arrival + assets in a single nested Prisma write.
- **Schema** (`src/schema/`): Zod validators for request bodies.
- **Middleware** (`src/middleware/`): Custom validation (barcode format, date range checks).

**Data access patterns:**
- Simple CRUD: Prisma ORM (`prisma.model.findMany()`, etc.)
- Complex reads: Typed raw SQL via `prisma.$queryRawTyped()` with `.sql` files in `prisma/sql/`. These are compile-time type-checked via Prisma's `typedSql` preview feature.
- Sequence numbers (barcodes, arrival numbers) are generated by calling `get_next_sequence(entity, warehouseCode, date)` — a PostgreSQL function that atomically increments a counter in the `Sequence` table.

**Barcode format:** `{warehouseCode}-{yyMMdd}-{0001}` (4-digit sequence)
**Arrival number format:** `A{warehouseCode}-{yyMMdd}-{001}` (3-digit sequence)

## Key domain concepts

- **Asset**: The core entity — a physical machine with a barcode, serial number, model, location, and three status axes: `TrackingStatus` (e.g. RECEIVING, IN_STOCK), `AvailabilityStatus` (e.g. AVAILABLE), and `TechnicalStatus`.
- **Arrival**: A batch of assets received from a vendor via a transporter, destined for a warehouse. Creates assets with initial meter readings and costs.
- **Departure / Transfer / Hold**: Other movement types. Departures ship to customers, transfers move between warehouses, holds reserve assets.
- **Location**: Scoped to a warehouse (composite key `warehouse_id + location`). Special locations include `ARRIVAL`, `TRANSFER`, etc.
- **Sequence table**: Tracks daily per-entity per-warehouse counters used for generating human-readable identifiers.

## Environment

Requires `DATABASE_URL` in `.env` (PostgreSQL connection string, currently Railway). Port defaults to `3000`.

CORS is restricted to `http://localhost:5173` (local Vue frontend) and `https://shiva-inv.vercel.app`.
